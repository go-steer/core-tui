# Gutter and Code Highlighting in Terminal Viewports

This document outlines the architectural patterns and implementations used to display and highlight inline file modifications (added, deleted, and unchanged lines) inside an interactive terminal user interface (TUI) viewport.

The design supports both **unified** (inline) and **split** (side-by-side) layouts. It highlights changed and new code by combining three core terminal UI techniques: **layout-based line styles**, **grapheme-width terminal viewport trimming**, and **integrated background syntax highlighting**.

---

## 1. Structural Line-Style Definitions
Line-level styles are managed through a structured hierarchy. Each line type (Equal, Insert, Delete) is styled using a dedicated style configuration containing three distinct visual targets:
1. **`LineNumber`**: Styles the gutter line number column.
2. **`Symbol`**: Styles the gutter change indicator (e.g., `+` or `-`).
3. **`Code`**: Styles the actual line of code.

These are configured with vibrant, custom dark-mode hex palettes:

```go
InsertLine: LineStyle{
	LineNumber: NewStyle().
		Foreground(colorGreenGutter).
		Background(Color("#293229")), // Dark green gutter background
	Symbol: NewStyle().
		Foreground(colorGreenSymbol).
		Background(Color("#303a30")), // Solid green indicator background
	Code: NewStyle().
		Foreground(colorTextLight).
		Background(Color("#303a30")), // Solid green code background
},
DeleteLine: LineStyle{
	LineNumber: NewStyle().
		Foreground(colorRedGutter).
		Background(Color("#332929")), // Dark red gutter background
	Symbol: NewStyle().
		Foreground(colorRedSymbol).
		Background(Color("#3a3030")), // Solid red indicator background
	Code: NewStyle().
		Foreground(colorTextLight).
		Background(Color("#3a3030")), // Solid red code background
}
```

---

## 2. Highlighting Layout Generation
During viewport rendering, the system iterates over differences generated by a diff engine. Depending on the line kind, it applies the relevant style:

* **Added Lines (Insert)**:
  * Renders a blank/space in the "before" line number gutter.
  * Formats the active "after" line number with the green `InsertLine.LineNumber` style.
  * Prepends a `+` symbol styled with green `InsertLine.Symbol`.
  * Renders the added code styled with `InsertLine.Code` (light text on a `#303a30` solid green background).
* **Removed Lines (Delete)**:
  * Formats the original "before" line number with the red `DeleteLine.LineNumber` style.
  * Leaves the "after" line number gutter blank.
  * Prepends a `-` symbol styled with red `DeleteLine.Symbol`.
  * Renders the deleted code styled with `DeleteLine.Code` (light text on a `#3a3030` solid red background).

---

## 3. Integrated Syntax Highlighting
Rather than rendering flat red or green code text, **the viewer applies full programming syntax highlighting on top of the diff backgrounds**:

1. During the line-rendering phase, the code content is processed alongside the specific line type's background color (e.g., green for insert, red for delete).
2. The tokenizer parses the source text, dynamically creating a syntax-highlighted formatter pre-configured with that background color:
   ```go
   f := getChromaFormatter(bgColor)
   ```
3. When the syntax tokenizer formats the code tokens (coloring keywords, strings, comments, and types), it forces the background of every token span to match the green or red background.
4. This results in syntax highlighting where the code’s syntax tokens remain fully colored but render seamlessly over the green or red diff-block backgrounds, ensuring readability and visual premium.

---

## 4. Caching & Performance Optimization
Because executing regular expression matching and syntax parsing over thousands of lines on every terminal render cycle would cause interface lag, the viewport utilizes a two-tier caching mechanism:
* **Lexer Cache**: It caches the file's matching language lexer to avoid repeating file extension pattern matching on every line.
* **Syntax Cache**: It caches the finalized, ANSI-escaped highlighted string in a local map keyed by the hash of the source code content and the active background color. This guarantees that scrolling through files in the terminal viewport occurs at a highly performant, lock-free framerate.
